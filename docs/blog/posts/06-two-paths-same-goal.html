Nikki at Transport Evolved retrofitted a JuiceBox with OpenEVSE hardware and Node-RED to enable load-sharing and smarter scheduling. I extended a CPH50 with a serverless script. Different hardware, different skills, same goal: make home charging smarter.

The Local vs. Cloud Debate

I'll admit it: I'm envious of Nikki's home-based system. There's something elegant about hardware running on your local network, independent of cloud infrastructure. No API dependencies, no third-party uptime concerns, no wondering if ChargePoint will rate-limit your polling. Just a Raspberry Pi, some Node-RED flows, and direct control over the pilot signal.

My path relies heavily on the cloud: GitHub Actions for scheduling, ChargePoint's API for control, and their servers to track my SDGE rate plan. It works‚Äîbut it's not autonomous in the way Nikki's system is.

The Trajectory: From Cloud to Local

Eventually, I'd love to migrate the entire charging automation to Home Assistant instead of relying on Google/GitHub infrastructure. The appeal is simple: plug things in, and they work. No third-party modules, no API wrapper libraries, no community forum digging.

But here's the thing: right now, the cloud approach solves my problem perfectly. ChargePoint already knows my SDGE rate schedule‚ÄîI don't have to maintain that logic. Their Energy Price Plan integration handles time-of-use complexity so I don't have to reverse-engineer rate tables or update my code when SDG&E changes pricing tiers.

Skill Profiles

  • Nikki: Hardware generalist, comfortable opening the EVSE, swapping boards, wiring CT clamps, using Node-RED for local flows.
  • Me: Software developer focused on modeling good practices‚Äîversion control, testing, documentation‚Äîfor students. Zero hardware changes, cloud automation.

Approach Trade-offs

Hardware Retrofit (Nikki's Path)

  • Gains: Deep control (pilot signal, per-phase monitoring), local network independence, no API rate limits
  • Costs: Requires safety diligence, physical modifications, hardware troubleshooting skills
  • Autonomy: Fully local‚Äîworks even if internet goes down

API-Layer Automation (My Current Path)

  • Gains: Non-invasive, leverages existing charger capabilities (like SDGE rate plan tracking), faster to deploy
  • Costs: Depends on reverse-engineered endpoints, vendor API uptime, cloud infrastructure reliability
  • Autonomy: Cloud-dependent (but aspiring toward local)

Why I Haven't Gone Local Yet

The honest answer? ChargePoint is doing work I don't want to maintain.

SDGE's time-of-use rates are complex: super off-peak (midnight-6am), off-peak (6am-4pm), peak (4pm-9pm), super off-peak again (9pm-midnight). Different rates for summer vs. winter. Occasional rate adjustments. ChargePoint's Energy Price Plan integration handles all of this‚Äîthey update their database when rates change, and my CPH50 just knows when to charge.

If I went fully local, I'd have to:

  • Scrape SDGE's rate schedule (or manually update a config file every time rates change)
  • Handle DST transitions correctly
  • Test edge cases (holidays, special events, SDG&E system outages)
  • Maintain this logic forever

That's a lot of undifferentiated heavy lifting for a problem ChargePoint has already solved.

The Path Forward: Hybrid Approach

I see my trajectory moving toward local control with selective cloud dependencies:

  1. Phase 1 (current): GitHub Actions + ChargePoint API‚Äîfully cloud-dependent
  2. Phase 2 (future): Evaluate plug-and-play local hardware options (no community forks, just install and go)
  3. Phase 3 (future): Migrate to Home Assistant for scheduling, but still use ChargePoint's rate plan tracking via their API
  4. Phase 4 (aspirational): Full local control with OpenEVSE hardware retrofit‚Äîbut only if ChargePoint discontinues their Energy Price Plan or their API becomes unreliable

The key insight: you don't have to choose between 100% local or 100% cloud. Use cloud services where they add value (like rate plan tracking), and own the critical path locally (like the 6am trigger logic).

Common Ground

  • Both lean on community knowledge‚ÄîOpenEVSE docs for Nikki; python-chargepoint and forum threads for me.
  • Both preserve something valuable: Nikki keeps her hardware investment; I keep Energy Price Plan scheduling.
  • Both add a safety valve: manual override still works (Charge Now in the app; physical controls on the EVSE).
  • Both recognize that the "perfect" solution is context-dependent‚Äîthere's no universal right answer.
There is no single "right" way to extend a charger. Your constraints‚Äîservice size, budget, trust model, skills, and whether your utility's rate plan is already integrated‚Äîdecide the path.

Why This Comparison Matters

It's tempting to claim one approach is superior. The reality: they're optimized for different constraints. Watching Nikki's build validated that DIY energy projects are thriving across hardware and software. It also reinforced my design choice: zero hardware (for now), keep the vendor scheduling, add one precise automation.

But I won't pretend I've reached the finish line. Someday, I might find a plug-and-play local solution that doesn't require community support overhead. Or I might open up that CPH50 and swap in OpenEVSE hardware. For now, though, the cloud path gets me 95% of the way there with 5% of the complexity.

And that's a trade-off I'm willing to make. For now.
